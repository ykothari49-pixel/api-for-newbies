Welcome

Enough about APIs to know what's happening — and where to look when it breaks

Built for product managers. You won't write code. You'll learn to read the conversation between systems, recognize the failure signatures, and ask engineering the right questions.

What you'll be able to do by the end

  • Read a log line or error and know roughly whose fault it is — the app, the server, or the network.
  • Understand what an engineer means by "the API returned a 500" or "it's a CORS issue."
  • Open your browser's Network tab during an incident and see the actual requests failing.
  • Tell the difference between "slow," "broken," and "rejected" — three very different bugs.
The one analogy to hold onto

Think of software systems as people in a giant office building who can only communicate by passing notes. An API is the set of rules for what a valid note looks like and what counts as a valid reply. Almost everything in this course is just: what's on the note, who it went to, and what came back.

We'll use fintech / banking examples throughout — balances, transactions, payments, live price feeds — since that's the world you'll recognize. Every demo on this site is simulated in your browser: nothing is sent over the internet, so you can click freely — except the final 🚀 Go Live module, where you'll send a real request to a real public API, on purpose. That's the graduation moment.

Foundations

How two systems talk: the request / response cycle

Every interaction you care about is one machine asking another machine for something, and getting an answer back. Master this shape and the rest is detail.

Client and server

Two roles in every conversation:

  • Client — the one who asks. Your banking app on a phone, a website in a browser, another internal service.
  • Server — the one who answers. The bank's computers that actually hold the account data.
Analogy: the bank teller

You (the client) walk up to a teller (the server) and say "What's the balance on account 4471?" The teller checks, and says "₹52,300." You asked a request; they gave a response. You don't get to walk behind the counter and rummage through the vault — you can only ask in the approved way. That approved way is the API.

The cycle, every single time

  1. Client sends a request — "I want the balance for account 4471."
  2. Request travels over the network — across wifi, cables, the internet.
  3. Server receives it, does work — checks you're allowed, looks up the data.
  4. Server sends a response — the balance, plus a status saying how it went.
  5. Response travels back — and your app shows it on screen.
Why this matters for debugging: when something breaks, it broke at one specific step. "Did the request even leave the phone?" "Did the server get it but reject it?" "Did the server answer but the answer was wrong?" These are different bugs with different owners. Most of this course is teaching you to locate the broken step.

Foundations

What is an API, really?

API = Application Programming Interface. Strip the jargon: it's a menu of things you're allowed to ask for, and the rules for asking.

Analogy: the restaurant menu

You don't walk into the kitchen and cook. You read a menu (the API) that lists what you can order and how to order it. You hand the waiter a structured request ("Item 12, no onions"). The kitchen (the server) does messy work you never see, and out comes a plate (the response). The menu is a contract: if it says "we serve pasta," you can rely on that. The API is that contract between two pieces of software.

Endpoints: the items on the menu

An API is made of endpoints — specific addresses you send requests to, each doing one job. They read like folders in a URL:

# A bank's API might expose endpoints like:
/accounts/4471            → details for one account
/accounts/4471/balance    → just the balance
/accounts/4471/transactions → the transaction history
/transfers                → where you'd POST a new money transfer
/fx/rates?from=USD&to=INR  → a live exchange rate
PM translation: when an engineer says "we need to add an endpoint," they mean "we need to build a new item on the menu — a new thing other software can ask our system to do." When they say "the endpoint is flaky," one specific item on the menu is misbehaving, not the whole system.

The full address: base URL + endpoint

https://api.examplebank.com  ← base URL (which server)
/accounts/4471/balance       ← endpoint path (what you want)
?currency=INR                ← query parameters (options/filters)

JSON: the language of the notes

Most modern APIs pass data as JSON — a simple text format of labels and values. You'll see it constantly. You don't need to write it, just read it:

{
  "accountId": "4471",
  "balance": 52300.50,
  "currency": "INR",
  "status": "active"
}

That's it — "label": value pairs, wrapped in braces. Curly braces {} = an object (one thing). Square brackets [] = a list of things. When a screen shows wrong data, an engineer will look at the raw JSON the server sent to see if the data was wrong or the app's display was wrong.

Mental model: An API is a contract + a menu. The contract says "ask me like this, I'll answer like that." When two teams integrate, they're agreeing on an API. When something breaks at the boundary between two systems, someone broke the contract — sent the wrong shape, or changed the menu without telling the other side.
The Mechanics

Anatomy of a request & response

Every note has the same four parts. Learn them once and you can read any request, in any tool, forever.

A request has four parts

PartWhat it isBank example
MethodThe verb — what action you wantGET (read), POST (create)
URLThe address — what you're acting on/accounts/4471/balance
HeadersMetadata about the request — who you are, what format you speakAuthorization: Bearer abc123
BodyThe payload — data you're sending (only for writes){ "amount": 500, "to": "8890" }
Analogy: a posted letter

Method = what you want done (deliver / pick up). URL = the address on the envelope. Headers = the postmarks, return address, "fragile" stickers — info about the letter. Body = the actual letter inside. A GET request is often an envelope with no letter inside — you're just asking to read something.

Headers — the part PMs underestimate

Headers carry the stuff that decides whether your request even gets accepted. The big ones:

  • Authorization — proves who you are (a token/key). Wrong or missing → 401. The #1 cause of "it works for me but not for them."
  • Content-Type — what format the body is in (usually application/json). Mismatch → the server can't parse your data → 400.
  • Accept — what format you want back.

A response has parts too

  • Status code — a 3-digit number saying how it went (next module — this is the single most useful thing to learn).
  • Headers — metadata about the response.
  • Body — the actual answer (the JSON data, or an error message).

The Mechanics

HTTP methods: the verbs

The method is the action you want. There are only a handful that matter. They map almost perfectly to "read, create, update, delete."

MethodMeansBank exampleChanges data?
GETRead / fetchGet account balanceNo — safe to retry
POSTCreate newMake a new transferYes — careful with retries
PUTReplace fullyOverwrite a saved payeeYes
PATCHUpdate partiallyChange just the payee's nicknameYes
DELETERemoveDelete a saved cardYes
Why this matters in incidents: A failed GET is annoying but usually safe — nothing changed, you can retry. A failed POST is scary: did the transfer go through or not? This is why you'll hear engineers obsess over "idempotency" — making it safe to retry a POST without charging the customer twice. If a customer says "I got charged twice," this is the area to look.
GET vs POST in one line

GET = asking a question (no side effects — ask it a hundred times, nothing changes). POST = signing a form (something happens in the real world — money moves, an account is created).

The Mechanics

Status codes: the most useful thing you'll learn

Every response carries a 3-digit code. The first digit tells you the category — and crucially, whose problem it probably is. Memorize the five families and you can triage almost any incident.

FamilyMeaningWhose fault, usually
2xx✅ Success — it workedNobody — celebrate
3xx↪️ Redirect — "go look over there instead"Usually fine, sometimes config
4xx🙅 Client error — you asked wrongThe caller (bad data, no auth, wrong URL)
5xx💥 Server error — they brokeThe server / backend team
The single most valuable distinction in this whole course: 4xx = the request was bad (the caller's fault — fix the request). 5xx = the server fell over (the backend's fault — the caller did nothing wrong). When triaging: a wall of 4xx means clients are doing something wrong (or a contract changed). A wall of 5xx means your backend is on fire — page the engineers.

Click any code to learn it

Memory hook: 200 = OK, 400 = "bad request, fix your note," 401 = "who are you?", 403 = "I know who you are, you're still not allowed," 404 = "that doesn't exist," 500 = "I broke," 503 = "I'm overwhelmed, try later." If you remember only these seven, you're ahead of most people in any incident channel.
The Mechanics

HTTPS & security: what the lock actually means

HTTP is the language. The S in HTTPS is the security layer wrapped around it. For a fintech product this is non-negotiable — but you only need the conceptual version.

HTTP vs HTTPS

  • HTTP — the notes are sent in plain text. Anyone between you and the server (on shared wifi, your ISP, etc.) can read them. Fine for a public webpage; catastrophic for a bank balance.
  • HTTPS — the same notes, but encrypted in transit. Eavesdroppers see only scrambled noise. The padlock in the browser = "this connection is encrypted and the server proved it's really examplebank.com."
Analogy: postcard vs sealed armored envelope

HTTP is a postcard — every postal worker along the way can read it. HTTPS is a tamper-proof, sealed envelope that only the recipient can open, plus a verified ID confirming the recipient is who they claim to be. For money, you always want the sealed envelope.

What HTTPS protects (and what it doesn't)

HTTPS DOESHTTPS does NOT
Encrypt data in transit (between client and server)Protect data once it's stored on the server
Verify you're talking to the real server (via a certificate)Mean the server's code is secure or bug-free
Stop eavesdropping & tampering on the networkReplace authentication — you still need login/tokens

Three terms you'll hear in security conversations

  • Certificate (SSL/TLS cert) — the server's verified ID card, issued by a trusted authority. "Certificate expired" is a real, common outage: the ID lapsed, so browsers refuse to trust the server and users see scary warnings. Someone forgot to renew it.
  • TLS — the modern protocol that does the actual encryption (SSL is its older name; people still say "SSL" loosely).
  • Token / API key — your proof of identity inside the encrypted channel (the Authorization header from earlier). HTTPS hides it from eavesdroppers; the token is what tells the server it's really you.
Incident you'll eventually see: "Users on iOS are getting security warnings and can't log in." Two prime suspects: (1) a TLS certificate expired or was misconfigured, or (2) the app is somehow hitting http:// instead of https:// and the OS is blocking it. Both are infrastructure issues, not app-logic bugs — useful to know so you route it to the right team fast.
Tools & Real-time

Reading curl (and why engineers paste it at you)

curl is a command-line tool for sending a single request by hand. You won't live in it, but engineers constantly share curl commands to say "here's the exact request that's failing." Being able to read one is a superpower in a debugging thread.

Anatomy of a curl command

It's just the four request parts from earlier, written on one line:

curl -X POST # the METHOD (-X = "use this method")
  https://api.examplebank.com/transfers # the URL
  -H "Authorization: Bearer abc123" # a HEADER (-H = header)
  -H "Content-Type: application/json" # another header
  -d '{ "from": "4471", "to": "8890", "amount": 500 }' # the BODY (-d = data)
FlagMeansMaps to
-XThe method to useMethod (GET/POST/…)
-HAdd a header (repeat for each)Headers
-dThe data / payload to sendBody
-i / -vShow response headers / verbose detailsDebugging output
Rule of thumb: no -X and no -d? It's a plain GET (just reading). See -d with a JSON blob? Something is being created or changed — look closely, especially for money.
Interactive: build a curl command

Adjust the pieces and watch the curl command assemble itself. This is exactly what an engineer would hand you.

Pro PM move: in an incident, ask the engineer "can you paste the curl that reproduces it?" Then you can see the exact endpoint, whether auth was included, and what data was sent — and often spot the problem yourself (wrong account ID, missing token, malformed body).
Tools & Real-time

WebSockets: when you need a live connection

Normal HTTP is one question, one answer, then hang up. Great for "what's my balance." Terrible for "show me the live USD/INR rate that changes every second." That's what WebSockets are for.

The core difference

HTTP (request/response)WebSocket
ShapeAsk → answer → done. Repeat for each update.Open one connection, leave it open, data flows both ways.
Who can talkOnly the client can start. Server can't push.Either side can send anytime (server can push to you).
Good forBalances, transaction history, settings — things you fetch on demand.Live prices, trading feeds, chat, notifications, "X is typing…".
Analogy: letters vs a phone call

HTTP is mailing letters — you send one, wait for the reply, send another. WebSocket is a phone call you leave connected — both of you can talk the moment there's something to say, no re-dialing. For a stock ticker, you don't want to mail a letter every second asking "price now? price now?" — you want the line open so the server just tells you the instant it changes.

"Polling" — the workaround you'll hear about

Before reaching for WebSockets, teams often poll: the app re-sends a normal HTTP request every few seconds asking "anything new?" It's simpler but wasteful (lots of "nope, nothing" replies) and laggy. If a PM asks "why is our price feed 5 seconds behind?" — the answer is often "we're polling every 5s; we'd need WebSockets for real-time."

Live FX feed — disconnected

A simulated live USD/INR rate feed. With HTTP you'd refresh manually; here the "server" pushes a new price whenever it moves. Connect and watch it stream.

—
Connection closed. Press "Open connection" to start the live feed.

Notice: once connected, you never ask again — the prices just arrive. That push-without-asking is the whole point of WebSockets.

How WebSockets break (and what you'll hear): "The feed froze" usually means the connection dropped silently and didn't reconnect — common on flaky mobile networks or after the phone sleeps. A robust client should auto-reconnect; if it doesn't, the screen shows stale prices with no error. So "frozen, but no error message" is a classic WebSocket signature — different from HTTP, which fails loudly with a status code.
Tools & Real-time

Webhooks: when the API calls you

Everything so far has been your system asking someone else's system. A webhook flips the direction: another system sends your server an HTTP request the moment something happens. It's how payment gateways tell you "the payment just succeeded."

The core idea

A webhook is just a normal POST request — same method, headers, JSON body you already know. The only new thing is who sends it: instead of you calling their API, you give them a URL on your server (e.g. /webhooks/payments), and they call it whenever an event happens.

Analogy: stop calling the courier — give them your doorbell

Polling is phoning the courier company every five minutes: "Has my package arrived yet? …Now? …Now?" A webhook is giving the courier your address and letting them ring your doorbell when the package actually shows up. You do nothing until the bell rings. The catch: if your doorbell is broken (your server is down), the courier may leave — and you'll never know they came.

Webhook vs API call vs WebSocket

API callWebSocketWebhook
DirectionYou → them, you askOpen pipe, both waysThem → you, they call your URL
WhenWhenever you want dataContinuous streamOnly when an event happens
Good forFetching on demandLive feeds (prices, chat)Notifications between servers: payment settled, KYC approved, refund processed
Bank exampleGET the balanceStreaming USD/INR ticksRazorpay/Stripe POSTs "payment.captured" to your server
PM translation: when an engineer says "we'll listen for the webhook," they mean: we won't keep asking the gateway if the payment cleared — the gateway will tell our server the instant it does. Checkout flows, subscription renewals, KYC status, chargebacks — in fintech, webhooks carry most of the "something just happened" traffic between companies.

How webhooks fail — the PM checklist

  • Missed deliveries. Your server was down when the webhook fired. Good senders retry (1 min, 5 min, 30 min…); bad ones don't. Classic symptom: "payment succeeded at the gateway, but the order is stuck at 'processing' on our side."
  • Duplicate deliveries. Retries mean the same event can arrive twice. Your side must handle a repeat without double-shipping the order — the same idempotency idea from the methods module, now on the receiving end.
  • No ordering guarantee. "payment.captured" can arrive before "payment.created." Code that assumes order breaks in weird ways.
  • Anyone can POST to a URL. If your endpoint blindly trusts whatever arrives, an attacker can fake a "payment succeeded" event. That's why webhooks carry a signature (a header proving the gateway really sent it) that your server must verify.
Incident you'll eventually see: "Customers paid but their orders never activated." First questions: Did the gateway send the webhook (check their dashboard — good providers show delivery logs)? Did our endpoint return 200 or error out? Are the retries exhausted? Can we replay the missed events? "Replay the webhooks" is a magic phrase worth knowing — most gateways can re-send past events.
Mental model: an API call is you asking; a webhook is being told. Both are the same HTTP notes you've already learned to read — only the arrow is reversed. And because the sender can't show an error on your screen, webhook failures are silent by default: the money moved, but nobody told your system.

When It Breaks

The troubleshooting playbook

When something's broken, you don't need to fix it — you need to localize it and route it. Here's the decision tree that gets you 80% of the way in any incident.

Step 1: Did the request even leave?

Open the Network tab (next module). If you see no request at all, the bug is in the app/frontend — a button isn't wired up, a condition is wrong. The backend is innocent.

Step 2: If it left, what came back?

2xx → it succeeded. If the screen is still wrong, the bug is in how the app displays the data, not the API. Check the response body vs what's shown.
401 / 403 → auth problem. Token missing, expired, or insufficient permissions. Extremely common. Route to: whoever owns login/sessions.
404 → wrong URL or the thing genuinely doesn't exist. Check the endpoint and the ID (e.g. typo'd account number).
400 / 422 → bad data in the request. The body is malformed or missing a required field. Often a contract mismatch after a change.
429 → rate limited. Too many requests too fast. The system is protecting itself. Back off / batch.
500 / 502 / 503 → the server broke or is overwhelmed. Not the caller's fault. Page the backend team — this is the "wake people up" category.
No response / timeout → network issue, server unreachable, or it's hung. Could be DNS, firewall, or the server is down entirely.

Step 3: The three flavors of "broken"

SymptomLikely categoryFirst question to ask
It's slow but worksPerformance — server slow, or polling, or big payloads"How long does the request take in the Network tab?"
It's rejected (4xx)The caller sent something wrong"What's in the request — auth, body, URL?"
It errors (5xx) or times outThe server/backend is failing"Is this happening for everyone, and when did it start?"

The questions that make you sound like you know your stuff

  • "What status code are we seeing?" — instantly narrows fault to client vs server.
  • "Is it all users or some?" — all = systemic (deploy, outage); some = data/account-specific or regional.
  • "When did it start, and did anything ship around then?" — most incidents trace to a recent change.
  • "Can you paste the curl that reproduces it?" — gives you the exact failing request.
  • "Is it the request being rejected or the response being wrong?" — separates 'won't go through' from 'wrong data.'"
Your job isn't to fix it — it's to localize and route. "We're seeing 503s across all users starting at 14:20, right after the payments deploy" is a sentence that gets the right people moving in seconds. That's the whole skill.

Where to look: the Network tab

Your browser has a built-in window into every request it makes. You already have it. Learning to open it and read it is the most practical skill in this course.

Opening it (do this now on any website)

  • Chrome / Edge: right-click → Inspect → click the Network tab. Or Cmd+Option+I (Mac) / F12 (Windows).
  • Safari: enable the Develop menu (Settings → Advanced → "Show features for web developers"), then Cmd+Option+I → Network.
  • Then reload the page or click around — you'll see every request stream in as rows.

Reading the rows — a simulated Network tab

This is what you'd see. Click any row to inspect it, just like the real thing.

NameMethodStatusTypeTime

The green row is healthy. The orange and red ones are the kind you'll hunt for in an incident — click them to see what's wrong and what you'd do.

What to actually click on, in order

  1. Status column — scan for anything not in the 200s. Red/orange = your suspects.
  2. The failing request → Headers — was the Authorization header there? Right URL?
  3. → Payload / Request — what data did we send? (catch wrong account IDs, missing fields)
  4. → Response / Preview — what did the server actually say? Error messages live here.
  5. Time column — anything taking seconds is a performance suspect.
One more place to look — the Console tab. Right next to Network. Red text there = JavaScript errors in the app itself (frontend bugs). If the Network tab shows all 200s but the page is broken, the Console usually has the answer. Two tabs, two halves of the story: Network = the conversation with the server, Console = the app talking to itself.
